# AIO POC Azure DevOps Backlog / User Story List

This phased backlog is intended to seed Azure DevOps epics/features/user stories for the POC implementation.

| Work Item   | Phase       | Feature                                   | User Story                                                                                                                                                     | Acceptance Criteria                                                                                                                 | Primary Team             |
|:------------|:------------|:------------------------------------------|:---------------------------------------------------------------------------------------------------------------------------------------------------------------|:------------------------------------------------------------------------------------------------------------------------------------|:-------------------------|
| Epic 1      | Phase 0-1   | POC foundation                            | As the platform team, we need a defined site boundary, landing zone, tags, and ownership model so the POC can be governed consistently.                        | POC scope approved; RG/subscription created; tagging standard applied; owner map documented.                                        | Platform                 |
| Epic 2      | Phase 2     | Arc-enabled K3s baseline                  | As the platform team, we need a hardened Arc-enabled K3s cluster with required Arc features so AIO can be deployed securely.                                   | Cluster Arc-enabled; OIDC/workload identity/custom locations/cluster connect enabled; secrets encryption and audit logging enabled. | Platform                 |
| Epic 3      | Phase 3-4   | Identity, PKI, and Key Vault baseline     | As the security team, we need workload identity, PKI decisions, and Key Vault secret delivery so workloads can authenticate without embedded credentials.      | UAMIs created; service accounts mapped; Key Vault RBAC enabled; private endpoint and DNS validated; CSI or SSE pattern selected.    | Security                 |
| Epic 4      | Phase 5     | Governance as code                        | As the platform and security teams, we need GitOps, Azure Policy, and Defender baselines so configuration drift is controlled and security posture is visible. | Flux configured; repos created; Policy extension installed in Audit mode; Defender for Containers enabled.                          | Platform/Security        |
| Epic 5      | Phase 6-7   | Network and observability baseline        | As the operations team, we need network segmentation and observability before go-live so failures can be detected and investigated.                            | Default-deny network policies in place; site egress approved; Monitor workspace, LAW, Grafana, and alerts deployed.                 | Platform/OT              |
| Epic 6      | Phase 8     | AIO secure deployment                     | As the app/platform teams, we need AIO deployed with secure settings so local edge services are available using the approved control plane.                    | AIO deployed; secure settings enabled; local-first validation completed.                                                            | Platform/App             |
| Epic 7      | Phase 9-10  | Cloud integration path                    | As the app team, we need eventing and storage services configured securely so curated edge data can flow to Azure.                                             | Event Hubs/Event Grid selected and configured; ADLS private endpoints and ACL model validated; diagnostics enabled.                 | App                      |
| Epic 8      | Phase 11-12 | Optional analytics and database workloads | As the app team, we need optional Databricks and one edge database pattern deployed securely to support the POC scenario.                                      | Databricks optional controls applied; PostgreSQL or MySQL deployed with TLS, namespace isolation, and backup validation.            | App                      |
| Epic 9      | Phase 13-14 | Recovery and operational readiness        | As all teams, we need recovery tests and runbooks completed so the POC can be operated and demonstrated with confidence.                                       | WAN outage simulation completed; restore tests completed; handoff runbooks published; access reviews scheduled.                     | Platform/OT/Security/App |

## Suggested decomposition guidance
- Use one Epic per phase/workstream.
- Break each Epic into Features aligned to the control areas in the implementation checklist.
- Track environment validation, evidence capture, and rollback tasks under each Feature.
- Attach architecture diagrams and source document references to each Epic/Feature as implementation notes.
